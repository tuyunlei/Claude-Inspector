# Claude Inspector – 产品 Vision 与 V1 战略切口

> 版本：v1 设计草案  
> 最近更新：请按当前日期填写

## 1. 产品整体愿景（Vision）

基于 `~/.claude` 工作区中的数据，为 **Claude Code** 的重度使用者提供可视化与分析能力，让他们可以：

- 更好地理解自己的 AI 编程 / 调试工作流；
- 更方便地复盘、讲述和分享这些工作流；
- 在需要时逐层下潜到真实的原始日志数据。

换句话说：**这是一个围绕「Claude 工作区」的观察与理解工具，而不是一个通用聊天产品。**

---

## 2. V1 战略切口：历史回放 & 复盘

在这个宽泛愿景之下，**V1 版本只专注做一件事：**

> 做一个专注于「历史回放 & 复盘」的工作区观察工具，  
> 让用户可以在一条跨 session 的时间线上，看清楚：  
> **我、Claude、subagent、tool 在什么时候做过什么事**，并且可以把这段 story 轻松导出分享。

目前优先服务的几个典型场景包括（但不限于）：

1. **给同事讲一个真实 case**  
   - 示例：分享「我如何用 Claude Code 搞定某次重构 / 调试」的全过程；
   - 需要从一堆零散的 session 中，整理出一条结构清晰、可展示的故事线。

2. **自己复盘一次长流程**  
   - 示例：回看「昨晚那次排查问题」的完整过程；
   - 关注点是：关键决策节点、各个工具和 subagent 的作用、有没有重复绕路。

3. **观察某类行为模式**  
   - 示例：只看某个 subagent（如 `refactor_agent`）在各个 session 中都干了什么；
   - 需要按「行为类型」而非单条 session 去聚合视角。

V1 不打算解决的事情（先刻意不做）：

- 不做「Claude 全能控制台」「实时 orchestrator」；
- 不把本工具变成一个通用 TODO 管理器或项目管理工具；
- 不以「配置管理」「系统健康监控」作为首要目标。

这些内容可以在未来版本中按模块化方式追加，但不应干扰 V1 对「历史回放 & 复盘」这一核心方向的打磨。

## 3. V1 设计原则（Design Principles）

下面这些原则用于指导后续的 IA / 交互 / UI 决策，任何新功能都应该能够清晰对应到其中至少一条，否则优先级需要重新审视。

### 3.1 Timeline first（时间线优先）

- 产品的主视图应该始终是一条「按时间排序的事件 / 会话时间线」；
- 不再以“左边 session 列表 + 右边聊天详情”的传统聊天窗口形态作为默认视图；
- 用户进入某个 project 时，第一眼看到的是「这段时间里发生了什么」，而不是一堆孤立的 session 卡片。

### 3.2 Cross-session by default（默认跨会话）

- 任何一次有意义的复盘或讲故事，往往会自然跨越多条 session；
- 用户不应该被迫记住「我刚才这个 case 分散在哪几条 session 里」并逐条点开对照；
- V1 的视图和交互，以**跨 session 的 story line** 为默认心智，而不仅是单条 session 的阅读器。

### 3.3 Actions are first-class（Claude 的“动作”是一等公民）

- tool 调用、subagent 执行、文件快照等事件，都是 Claude 的「实际动作」，应该在 UI 中显著突出；
- 这些 action event 应该有清晰的视觉样式与图标，并支持按类型过滤 / 聚合；
- 纯文本对话是上下文，而这些动作才是 story 的关键骨架。

### 3.4 From overview to raw logs（从概览到原始日志，可逐层下潜）

- 默认视图应该是「适合人脑阅读的故事线」：经过分组、折叠和精炼；
- 用户可以随时一键跳转到某个事件背后的原始 .json / .log 数据；
- 不牺牲可追溯性：任何 UI 层面的美化都不应该遮挡掉底层真实数据。

### 3.5 Everything worth seeing is exportable（重要视图都能带走）

- 用户在时间线上选定的一段 story（某个时间区间、若干关键事件），应能轻松导出成：
  - 长截图；
  - Markdown 文档；
  - 或其他适合演示 / 汇报的格式。
- 设计导出的内容结构时，以「讲给同事听」的场景为核心：  
  标题清晰、步骤分明、可以直接贴进文档或 slides 里使用。

## 4. 对后续设计 / 开发决策的影响

在讨论新的信息结构、交互和 UI 时，可以用以下问题进行自检：

1. 这个改动是否让用户更容易在「时间线」上看清 story？（对应 3.1 / 3.2）
2. 它是否帮助用户更好地理解 Claude 的动作，而不仅是看到更多对话？（对应 3.3）
3. 是否仍然保留了向下钻取到原始日志的路径？（对应 3.4）
4. 这块视图 / 功能是否有明确的导出或复用价值？（对应 3.5）
5. 它是否违背了 V1“只专注历史回放 & 复盘”的切口？如果是，应该推迟到未来版本。

只要一个改动不能在这里找到合理的位置，要么是还没想清楚问题，要么就是不应该出现在 V1 里。
